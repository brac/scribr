## What shipped

The `>20MB` path is real. A `POST /v1/validate` with a `blob_id` over 20MB no longer returns the temporary M3 `413` — it returns `202` + the SPEC_02 job envelope (`{ job_id, status: "queued", poll, result: null }`), charges credits up front, and hands the job off behind a one-method **JobQueue seam**. The report is produced by the same wasm core, stored the same way, and re-fetchable at `/v1/reports/:id`; the difference is purely *when* and *where* it runs.

New surface under `api/src/`:
- `jobs.ts` — the job types (`JobSpec`, `JobRunner`, `JobStatus`), the shared lifecycle operations (`insertQueuedJob`, `transitionToProcessing`, `runValidateAndStore`, `failJob` with the refund ruling, `processValidateJob`), and `renderJobEnvelope` (the verbatim-splicing envelope builder).
- `runners/inline.ts` — `InlineJobRunner`: runs the shared lifecycle in-process, with `defer` (queue + `runAll()`) and `haltAfterProcessing` (crash simulation) knobs for tests.
- `runners/workflow.ts` — `WorkflowJobRunner`: a thin `"use workflow"` wrapper (`start(processValidateWorkflow, [spec])`) whose `"use step"` functions call the same shared operations. Compile-ready against `workflow@4.6.0`, exercised live at P1M5.
- `webhooks.ts` — `assertPublicWebhookUrl` (SSRF ⇒ BAD_REQUEST at submit), `attemptWebhookDelivery` (one signed POST), and `deliverWebhookWithRetries` (3 attempts, injected backoff, swallow-on-exhaust).
- `sweep.ts` — `runSweep`: the four-pass retention sweep (expired reports, stuck jobs, orphaned uploads, old job rows).
- `routes/jobs.ts` — `GET /v1/jobs/:id` (own-account only; `JOB_NOT_FOUND` otherwise; verbatim report on `done`).
- `routes/cron-sweep.ts` — `GET /api/cron/sweep` (outside /v1; CRON_SECRET Bearer).

`Deps` gained `jobs: JobRunner` and an optional `cronSecret`; `CryptoProvider` gained `hmacSha256Hex`. Accounts gained a `webhook_secret` (migration `0003_webhook_secret.sql`, generated at creation in the admin CLI and the test harness, surfaced in `GET /v1/account`). `vercel.json` was created with the nightly cron plus Fluid `maxDuration` + `includeFiles` for the wasm pkg and config TOMLs.

Twenty-one new Vitest tests (101 total across 14 files) cover: the full job lifecycle (queued/processing/done/failed via the deferred and halt runners); the **envelope byte-pin** (embedded report === stored blob === direct wasm run); the refund ruling both ways (blob-store outage refunds, MALFORMED_GLTF does not); `JOB_NOT_FOUND` for a bad id and cross-account; webhook signing (HMAC recomputed from scratch), 3-retry-then-give-up (job stays `done`), retry-recovers-on-2nd, and private-webhook-rejected-pre-charge; the sweep's four passes under the injected clock; and the no-zombie recovery. `JOB_NOT_FOUND` is now reachable — **14 of 16** SPEC_03 codes (only `RENDER_FAILED` and `INSPECT_UNAVAILABLE` remain, both later phases).

## Decisions

The load-bearing decision is the **thin JobQueue seam with one shared lifecycle**. `processValidateJob` — mark processing, run + store, deliver webhook, clean up — is the single definition of what a job *does*. `InlineJobRunner` runs it directly; `WorkflowJobRunner`'s durable steps call the same operations. That is what lets the entire async path — including refunds, webhook retries, and no-zombie recovery — be exercised hermetically in Vitest while the production runner stays a compile-checked, inspection-reviewed artifact for P1M5.

**Charge before the row.** The submit path charges, *then* inserts the `queued` row, *then* enqueues. So a `402` never leaves a job row behind, and — the important invariant — a job row that exists was definitely charged. That is exactly what makes the sweep's stuck-job refund sound: every non-terminal row it fails-out has a credit to give back.

**Idempotency is encoded in guards, not hope.** `transitionToProcessing` only advances non-terminal rows; the store-then-flip is guarded `status <> 'done'`; `failJob` refuses to touch a `done`/`failed` row (so it can neither double-refund nor clobber a completed job); `reports_index` inserts are `ON CONFLICT DO NOTHING`. A step re-run after a crash — even one whose upload blob is already gone — leaves a `done` job `done` and issues no spurious refund. A test drives the whole pipeline twice to prove it.

**The verbatim discipline reaches the envelope.** `renderJobEnvelope` builds the `done` response as a string and splices the stored report bytes between `"result":` and the closing brace. No `JSON.parse`, no re-stringify — the report portion is byte-identical to what `/validate` and `/reports/:id` return. The `failed` and `queued`/`processing` branches carry a structured error / `null`, which are our own data and may be serialized normally.

## What broke

- **My first idempotency test found a real clobber bug.** I wrote a test that runs the whole job pipeline twice. The second run re-reads the upload blob — which the first run had already deleted — so `runValidateAndStore` threw, `failJob` fired, and its `WHERE status <> 'failed'` guard happily flipped the already-`done` job to `failed` **and refunded a credit**. That is a re-run corrupting a completed job. The fix was to tighten the guard to `status NOT IN ('done','failed')`, so `failJob` can only act on a non-terminal row. Now a re-run after completion is genuinely a no-op — which is the property the plan asked me to guarantee, surfaced by the test I almost didn't write.
- **Review caught the mirror-image race I missed: the done-transition could resurrect a swept-failed job.** Sequence: a job runs long → the sweep fails it out at the 1h timeout AND refunds → the slow worker step finally completes → the done-flip, guarded only against `done`, flips `failed`→`done` — report delivered, webhook fired, refund kept, a terminal state reversed. Fix: the done-transition guard became `status NOT IN ('done','failed')` (terminal is terminal, mirroring failJob), `runValidateAndStore` now returns `{ reportString, transitioned }` via `RETURNING`, and the webhook only fires when this run actually won the transition. The late-stored report blob is left in place deliberately (a valid paid-then-refunded artifact; retention ages it out) while the job row keeps `result_blob_key` NULL. A dedicated race test — job to `processing`, clock past the timeout, sweep fails-out + refunds, deferred runner completes late — pins status-stays-failed, zero webhook calls, refund stands, `result_blob_key` NULL.
- **The `workflow` package wasn't installed, and "compile-ready" means it has to typecheck.** The plan says pin it and keep the seam thin, but there was no dependency. I pinned and installed `workflow@4.6.0` (stable `latest`; `beta` is `5.0.0-beta.33`). The import surface I use is tiny — `start` from `workflow/api`, `RetryableError` from `workflow`, and the `"use workflow"`/`"use step"` directives, which are plain string literals to `tsc`. The runner lives in its own module that the test path never imports at runtime, so installing the package added a real typecheck target without dragging the WDK runtime into the suite.
- **Runner ↔ Deps is a cycle.** The route calls `deps.jobs.enqueue(spec)`, and the runner needs `deps` (db/blob/crypto) to run the job. I broke it with a thunk: `new InlineJobRunner(() => deps, opts)` reads `deps` lazily, and the harness assigns `deps` (with `jobs: runner`) immediately after — the thunk is only invoked at enqueue time, long after `deps` exists.
- **`webhook_secret` NOT NULL vs a populated table.** `drizzle-kit generate` emitted a bare `ADD COLUMN ... NOT NULL`, which would fail on any DB that already had accounts. I rewrote the migration to add the column nullable, backfill legacy rows with `md5(random()::text || id)` (no pgcrypto — runs identically on PGlite and Neon), then `SET NOT NULL`, so the end state matches the schema snapshot on both a fresh and a populated database.
- **Orphaned-upload reclaim has no blob↔job link, by design.** The job spec (which carries the `blob_id`) lives in-process for the inline runner and in the WDK event log for the workflow runner — never in a queryable column. So the sweep can't ask "is this upload blob referenced by a live job?". I made the reclaim purely age-based: a completed job deletes its own upload blob within minutes, so any `uploads/` blob older than 24h is by definition unclaimed. It is documented in `sweep.ts` as an intentional heuristic, not an oversight.

Known weaknesses, deliberately deferred:
- **The WorkflowJobRunner is inspection-verified, not run.** It typechecks against the real package and mirrors the shared lifecycle step-for-step, but no WDK runtime, Neon, or Blob has executed it. The system-fault-retry-then-sweep interplay in particular (WDK retries a `RetryableError` step; if it exhausts, the sweep recovers) is reasoned about, not observed. That is the explicit P1M5 gate.
- **Async still buffers the whole file before enqueue.** `acquireInput` reads the full blob into memory even for the async path (the P1M3 streaming weakness, unchanged) — the async trigger just routes on the already-loaded size. Streaming the model straight to the worker is P1M5, when the real Blob client-upload flow lands.
- **Webhook backoff is real time in production.** The inline runner injects an immediate wait for tests; the production/inline-local path uses real `setTimeout` backoff. The durable-step variant would let WDK own the retry cadence — that path is the compile-ready one, exercised at P1M5.

## Numbers

Measured on the dev machine (Windows, Node 22, pnpm 9.15 via corepack). `pnpm -r typecheck` (strict `tsc --noEmit`, `exactOptionalPropertyTypes`) is clean, including the `WorkflowJobRunner` compiled against `workflow@4.6.0`. `pnpm -r test` runs 101 Vitest tests across 14 files in ~13s, each suite spinning up its own in-memory PGlite (migrations + plan seed) and an in-memory BlobStore, with the InlineJobRunner driving the async lifecycle under a pinned clock — no Vercel account, no WDK runtime, no network. The job-envelope byte-pin holds: the report spliced into a `done` `GET /v1/jobs/:id` response is byte-identical to the stored blob and to a fresh wasm run of `Box.glb`. Fourteen of the sixteen SPEC_03 error codes are now reachable through `app.fetch` — `JOB_NOT_FOUND` joined the set, leaving only `RENDER_FAILED` (Phase 2 renderer) and `INSPECT_UNAVAILABLE` (vision). The Rust gates are untouched: `cargo test --workspace` is still 214 passed / 3 ignored, exactly as at p1m1–p1m3.

## Next

P1M5 provisions the real platform: a Neon branch, real Vercel Blob (`access:'private'`, the client-upload token flow), and the first LIVE exercise of the `WorkflowJobRunner` and `vercel.json` — deploying the workflow, running a `>20MB` job end-to-end through WDK, watching a step redeploy-retry, firing a real signed webhook, and letting the nightly cron sweep run against real storage. That is where the socket-level SSRF pin and the streamed upload path also land. After that the async foundation is production-real, and Phase 2 can add the renderer (`RENDER_FAILED`) on top of it.
